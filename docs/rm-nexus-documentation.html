RM Nexus Documentation

An AI-assisted "Next Best Conversation" dashboard for institutional banking Relationship Managers — what it does, how to run it, how it's built, and how it's governed. Company names are real; all financial data is synthetic.

01 · Overview & use case

What RM Nexus is, and why it helps

A Relationship Manager (RM) at an institutional bank looks after a book of large clients — super funds, sovereign wealth funds, asset managers, corporates. Their recurring question is simple to state and hard to answer well: "what should I talk to each client about next?" RM Nexus answers it. It renders the RM's book of 8 clients as scannable tiles and, for each, a prioritised Next Best Conversation (NBC) briefing — the three most worthwhile things to raise, most urgent first, each grounded in that client's actual position and the relevant APRA thresholds.

The benefit

Faster triage

The whole book is legible at a glance — alert-tinted tiles, live cash-flow sparklines, and a one-line "why now" per client — so an RM spots what needs attention without opening a spreadsheet.

Trustworthy by design

Every figure comes from the client dataset — never the AI. The model only narrates and prioritises; it cannot compute, round, or invent a number. No hallucinated figure can reach an RM as fact.

Always available

The dashboard works fully with no AI at all. A fresh AI briefing is an on-demand overlay; if it's unavailable, the card silently keeps its static briefing. No outage, no broken screen.

Built for a regulated shop

Human-gated change control, automated tests, and a governance & security review mapped to APRA's AI Letter, CPS 230 and CPS 234 — credible inside a bank, not just a demo.

The core idea in one line: data is authoritative; the LLM only narrates. That single boundary is what makes an AI feature safe to put in front of a banker.

02 · Using the dashboard

How to use it

  1. Scan the book

    Open the dashboard at /rm. Each of the 8 client tiles shows net position, total exposure, liquidity score, revenue & YoY, a 30-day cash-flow sparkline (green above the line = inflow, red below = outflow), product chips, and a colour tint keyed to the client's alert level (red = critical, amber = high, blue = medium, green = none).

  2. Read the static briefing

    Every tile always shows a briefing summary and a Show 3 NBC ▾ toggle that expands the three prioritised conversations — urgency, a short action, the reasoning (citing A$ figures and APRA APS 210 thresholds), and suggested next steps. This is the dependable floor: it needs no AI.

  3. Open a client for the full picture

    Click a tile (look for "View details ›") to open the full-screen client view: profile and contacts, currency mix, a 0–100 liquidity bar, a large cash-flow sparkline, product breakdown, the alert detail, and the AI NBC panel.

  4. Generate fresh AI briefings (optional)

    Click Generate All Briefings (dashboard) or Generate ▸ (a single client) to overlay fresh AI briefings. They stream in progressively, one client at a time. If a live model isn't configured, you'll see a deterministic template briefing instead — the system never breaks.

  5. Open this documentation any time

    The Documentation ↗ link in the top-right of the app nav opens this page.

03 · Launch & run

Running RM Nexus locally

Prerequisites: Python 3.10+ and Node.js (for Next.js 16 / React 19). No API key is required — that's the point.

Two servers

# Terminal 1 — backend (FastAPI) on :8000
python backend/run.py

# Terminal 2 — frontend (Next.js) on :3000
cd frontend
npm install      # first run only
npm run dev

Then open http://localhost:3000/rm.

Zero-live-LLM floor. With no ANTHROPIC_API_KEY set, the dashboard loads all 8 clients with their static briefings and makes zero AI requests — verified by a cold walk-away check. To enable live AI briefings, set ANTHROPIC_API_KEY (copy backend/.env.example to backend/.env); optionally override the model with ANTHROPIC_MODEL (default claude-haiku-4-5-20251001).

Verify it works

# Backend guarantees — key-free, network-free
cd backend && pytest -q            # 6 tests: boundary, no-key fallback, contract, SSE

# Browser visual checks — real Chromium, 5 spec scenarios
cd frontend && npm run e2e          # headless
cd frontend && npm run e2e:headed   # watch it drive a real browser

In this repo's Claude Code setup, the /launch skill starts both servers and opens the browser, and /verify runs the headed visual suite.

04 · Architecture

How it fits together

A Next.js frontend renders the book from a bundled dataset and, on demand, streams fresh AI briefings from a Python/FastAPI backend over Server-Sent Events. The AI provider is an optional overlay, drawn with a dashed line below to signal that nothing on the critical path depends on it.

The two boundaries that define the design

Data authority
The agent is tool-less (NBCAgent.tools = []) so it cannot compute or alter a figure, and NBCItem is all-strings (no numeric field) so a narrated number is never trusted as UI data. Two halves of one boundary.
Zero-LLM floor
Static "yesterday's briefing" per client; the live overlay degrades silently to it via the __NBC_ERROR__ path on any error, missing key, timeout, or malformed output.
Progressive reveal
A worker thread parallelises the blocking SDK calls and hands off to the async SSE endpoint through a stdlib queue.Queue, whose get-timeout doubles as the ~10 s keepalive tick.

05 · CI, controls & benefits

Continuous integration & change controls

From item 6 onward, main is protected: no change lands without a pull request, passing CI, and a human merge. The pipeline runs on every push and pull request, and every UI change carries browser screenshots as pre-merge evidence.

Controls and why they matter

ControlWhat it doesBenefit
Branch-protection ruleset (human-review-main) Requires a PR and passing backend/frontend checks; no bypass for anyone, including admins. Nothing reaches production without automated verification and a human decision — the AI cannot self-approve its own work.
git_guard hook A local PreToolUse hook that blocks direct push/commit to main during an agent session. Defence in depth — a local belt to the GitHub ruleset's suspenders, catching mistakes before they even reach the server.
Backend tests (pytest + ruff) Key-free, network-free tests assert the data-authoritative boundary and the no-key fallback on every push/PR. The two product guarantees can't silently regress — they're machine-checked, not just documented.
Visual verification (Playwright) Drives real Chromium against a cold, frontend-only build; asserts each UI criterion and captures a screenshot. A green run is also proof of the zero-LLM floor; assurance happens before merge, not after deployment.
Inline screenshot evidence Screenshots are published to a branch and embedded in a PR comment for the reviewer. A human can visually confirm the outcome matches intent, right in the PR — no local checkout needed.
Spec-driven trail (specs/) Each change references a spec and adds a README Features-log entry. Auditable "why", not just "what" — the change history reads as a record, not a pile of diffs.

10 · Glossary

Terms

NBC
Next Best Conversation — the prioritised set of things an RM should raise with a client.
RM
Relationship Manager — the banker who owns the client relationship.
APRA
Australian Prudential Regulation Authority — the regulator for AU banks, insurers and super.
CPS 230
APRA's Operational Risk Management standard (in force 1 Jul 2025) — critical operations, tolerance, service providers.
CPS 234
APRA's Information Security standard — controls, testing, and 72-hour incident notification.
APS 210
APRA's liquidity standard; the dashboard cites its warning thresholds where relevant.
SSE
Server-Sent Events — a one-way HTTP stream used to reveal AI briefings progressively.
Zero-LLM floor
The guarantee that the app is fully usable with no AI service available.

11 · Troubleshooting

Common questions

The briefings look generic / say the same structure — is it broken?

No. With no ANTHROPIC_API_KEY, "Generate" returns a deterministic template briefing built from the client's figures. That's the zero-LLM floor working. Set a key for live model narration.

Screenshots on this page don't load.

They're external references to the repository's evidence branch; you're likely offline or the repo is private to you. The rest of the page is self-contained.

The "Documentation" link 404s.

The app serves this file from public/, which is generated from docs/rm-nexus-documentation.html on npm run dev/build. If you opened the app another way, restart the dev server so the copy step runs.

Nothing streams when I click Generate.

Confirm the backend is up (curl http://localhost:8000/api/health → {"status":"ok"}). The browser connects directly to :8000 for the stream; if the backend is down, cards keep their static briefings by design.
